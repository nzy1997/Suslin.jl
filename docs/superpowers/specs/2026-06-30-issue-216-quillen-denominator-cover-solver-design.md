# Issue 216 Quillen Denominator Cover Solver Design

## Context

Issue #216 follows the #215 denominator-cover candidate extraction layer. The
repository already has a replayable `QuillenDenominatorCoverCertificate` that
accepts ordinary exact polynomial-ring denominators and coverage multipliers
when `sum(g_i * d_i) == 1`, and #215 now records raw local support
denominators `r_i` without proving the Park-Woodburn cover identity.

The missing stage is a bounded exact solver that takes raw denominators from a
`QuillenDenominatorCoverCandidate`, chooses an exponent `l`, proves
`sum(g_i * r_i^l) == 1`, and stores a result that keeps raw denominators
separate from the powered cover terms. Later substitution-chain code must not
accidentally exponentiate already-powered certificate denominators.

## Approaches Considered

Recommended: add an internal solver result in `src/algorithm/quillen_induction.jl`
and use Oscar's exact `coordinates(one(R), ideal(R, powered_terms))` to search
deterministically over `l = 1:max_exponent`. This reuses the same ideal
membership primitive already used by existing ECP tests, keeps the solver
bounded, and gives exact multipliers when the powered denominators generate the
unit ideal.

Alternative: add a reusable helper in `src/core/groebner_tools.jl`. This could
be useful later, but the current behavior is Quillen-specific because it must
package raw denominators, powered terms, candidate provenance, and the existing
Quillen cover certificate. A core helper would add an abstraction before there
is another consumer.

Alternative: accept only supplied multipliers and do no automatic ideal
membership. This would verify stale or tampered multipliers, but it would not
meet the issue objective because the current verifier would still not discover
cover terms.

The chosen design is the Oscar-coordinate solver in `quillen_induction.jl`.
It is deterministic, bounded, exact, and minimal.

## Solver Shape

Add two internal records:

- `QuillenDenominatorCoverSolverVerification`: replay data for a solver result,
  including raw denominators, exponent, powered denominators, multipliers,
  exact terms, coverage sum, candidate consistency, certificate replay, and
  overall status.
- `QuillenDenominatorCoverSolverResult`: the replayable solver output. It
  stores the source candidate when present, ring, raw denominators, chosen
  exponent, powered denominators, coverage multipliers, exact coverage terms,
  coverage sum, the existing `QuillenDenominatorCoverCertificate`, and the
  solver verification record.

Add these functions:

- `solve_quillen_denominator_cover(candidate; max_exponent = 4, exponent = nothing, coverage_multipliers = nothing, supplied_multipliers = nothing)`.
- `solve_quillen_denominator_cover(R, raw_denominators; max_exponent = 4, exponent = nothing, coverage_multipliers = nothing, supplied_multipliers = nothing)`.
- `replay_quillen_denominator_cover_solver_result(result)`.
- `verify_quillen_denominator_cover_solver_result(result)::Bool`.

The direct `R, raw_denominators` overload supports focused negative controls
such as `[r, s]` in `QQ[X,r,s]`. The candidate overload verifies the candidate
first and then solves its extracted raw denominators.

## Solver Semantics

Construction normalizes raw denominators and multipliers into the exact
ordinary polynomial ring using the same rules as the existing cover verifier.
It rejects an empty raw-denominator list, nonpositive `max_exponent`, and an
explicit exponent outside `1:max_exponent`.

Without supplied multipliers, the solver tries each requested exponent in
ascending order. For a fixed `l`, it builds `powered_terms = [r_i^l]` and asks
Oscar for coordinates of `1` in the ideal generated by those powered terms. If
coordinates are found, it replays the exact identity through
`quillen_denominator_cover_certificate(R, powered_terms, multipliers)` and
returns the result.

With supplied multipliers, the solver still normalizes and replays the exact
identity. If `exponent` is supplied, only that exponent is checked. If no
explicit exponent is supplied, the same bounded exponent sequence is checked
with the supplied multipliers. Count mismatches or failed identities throw
`ArgumentError` messages containing `coverage not proven`.

If no exponent proves coverage within the bound, the solver throws a deliberate
`ArgumentError` whose message contains `coverage not proven`. This is the
staged failure for later route-selection code.

## Replay And Tamper Checks

`replay_quillen_denominator_cover_solver_result` recomputes powered terms from
stored raw denominators and the stored exponent, recomputes exact coverage
terms and sum, verifies the stored source candidate when present, and checks
that the stored cover certificate is exactly the adapter for the powered terms
and multipliers.

`verify_quillen_denominator_cover_solver_result` compares the stored result and
stored verification against replay. Manual edits to a multiplier, exponent,
powered denominator, coverage sum, or certificate make verification return
`false`. The existing `verify_quillen_denominator_cover` remains the adapter
check for the certificate itself.
