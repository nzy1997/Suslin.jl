# Issue 9 Laurent Linear Solving Ownership Design

## Context

Suslin now has Laurent ring constructors, strict Laurent validators, shared
Laurent fixtures, and Laurent normalization helpers. Later Quillen patching and
column-reduction steps need exact solutions of systems `A * U = B` over one
Laurent polynomial ring.

ToricBuilder already has a working `solve_laurent_linear` implementation in
`src/core/laurent_linear_solve.jl`. Issue #9 asks Suslin to make an ownership
decision instead of blindly adding a second unrelated solver.

## Design Choice

Suslin will own a narrow Laurent linear solver by adapting the ToricBuilder
implementation into `src/core/laurent_linear_solve.jl` with focused tests.

The public function is:

```julia
solve_laurent_linear(A, B)
```

It accepts an `m x n` Oscar matrix `A` and an `m x k` Oscar matrix `B` over the
same Laurent polynomial ring, and returns an `n x k` matrix `U` satisfying
`A * U == B`.

Alternatives considered:

- Keep ToricBuilder as the owner and add only a wrapper. This would leave
  Suslin patching dependent on a non-package local checkout and would not give
  Suslin focused regression tests for the exact error contract it needs.
- Factor shared logic into a reusable helper layer. That may become attractive
  later, but it would broaden this issue into cross-repository packaging before
  Suslin has a stable solver boundary.
- Deliberately defer Laurent solving with staged errors. That would keep the
  roadmap blocked at the first Quillen patching step that needs an exact
  solution matrix.

The chosen model is the narrowest implementation that gives Suslin a stable
contract while preserving ToricBuilder reference behavior.

## Solver Semantics

Input validation happens before any solve attempt:

- `base_ring(A)` must be a Laurent polynomial ring.
- `base_ring(B)` must be the same Laurent ring object as `base_ring(A)`.
- `nrows(B)` must equal `nrows(A)`.

The solver first calls Oscar's native right-side solver:

```julia
can_solve_with_solution(A, B; side=:right)
```

If the native path reports no solution, `solve_laurent_linear` throws:

```julia
ErrorException("No exact solution exists for A * U = B")
```

If the native path fails with the known Laurent capability errors from Oscar's
`can_solve_with_solution` stack, Suslin falls back to the ToricBuilder
module-based approach. The fallback constructs the module generated by columns
of `A` in Oscar's polynomial quotient model for the Laurent ring, solves each
right-hand side column by module coordinates, maps the coefficients back to the
Laurent ring, and horizontally concatenates multi-RHS solutions.

Backend capability failures are therefore separated from mathematical
no-solution failures.

## Files

- Create `src/core/laurent_linear_solve.jl`: Suslin-owned solver, validation,
  native path, capability-error detection, and module fallback.
- Modify `src/Suslin.jl`: include the new core file and export
  `solve_laurent_linear`.
- Create `test/internal/laurent_linear_solve.jl`: focused tests for single-RHS
  exact solving, multi-RHS exact solving, no-solution behavior, mismatched
  parent rejection, and row-dimension rejection.
- Modify `test/runtests.jl`: include the focused test in the default internal
  group.
- Modify `test/public/api_surface.jl`: cover the new exported public function.
